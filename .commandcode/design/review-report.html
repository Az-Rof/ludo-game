// TL;DR

The palette refresh moved from generic dark-glass to a warm game table, and that was the right direction. The lobby and waiting room feel inviting. But the game screen — where players actually spend their time — still reads like a web form with a board embedded. Token movement has zero animation, async actions show no loading state, and the typography is clean but anonymous. The interaction layer needs to catch up to the visual mood.

Fix the game surface first. Token animation, loading states, and the mobile game screen composition will have the highest impact on how the game feels.

// WHAT'S WORKING

Palette Direction

Warm cream background with dark brown text breaks from the dark-mode reflex. Screen-specific accent colors (amber lobby, blue waiting) show intentional mood mapping.

Player Color System

Four game colors used as first-class design tokens unify the board and UI. Status inheriting the current player's color is a nice touch of system thinking.

Dice Interaction

Hover scale, active scale-down, and a spring-cubic-bezier roll animation give the dice physical weight. Border + shadow treatment reinforces the tactile feel.

// PRIORITY ISSUES
P0

Token Movement Has No Animation

animateMultiplayerMove() in ui.js just calls renderBoard() — tokens disappear and reappear at new positions. A board game about moving pieces has invisible movement. This is the single biggest UX gap.

FIX: /design motion — add step-by-step token animation that slides tokens along the track path before snapping to final position. 200ms per step.
P0

No Loading States on Async Actions

createRoom(), joinRoom(), startGame(), and addBot() fire socket events with zero visual feedback. Buttons stay idle. Users can click multiple times. No spinner, no disabled state change, no status text.

FIX: /design interaction — disable buttons on click, show spinner or text change, re-enable on response or timeout.
P1

Game Screen Vertical Scroll Problem

On mobile (≤600px), the game screen stacks 6 sections vertically: player bar → board → tile editor → dice → status → chat. The board dominates the viewport, pushing chat far below. Players can't see chat and the board simultaneously.

FIX: /design refine — collapsible chat, tabbed layout, or an overlay panel. At minimum, make chat toggleable.
P1

Typography Has No Game Voice

Font stack is 'Segoe UI', Tahoma, Geneva, Verdana, sans-serif — the Windows default. The lobby title at 3.5rem/800 weight is strong, but everything else is clean/system. For a product called "LUDO," the type carries no energy or playfulness.

FIX: /design typeset — introduce a display typeface for the game title and key headings. System fonts are fine for body/UI.
P1

Victory Pulse Animation Runs Indefinitely

pulse-victory runs with infinite iteration. A pulsing trophy emoji is celebratory for 2 seconds, then distracting. No user control to stop it.

FIX: Limit animation to 3-4 cycles, then settle with forwards fill.
P2

Amber Accent Overuse

Amber appears on: lobby top border, primary buttons, button hover shadow, input focus ring, player-btn selected state, room-code highlight, active player-indicator, dice value text, selectable token ring, chat send button, victory top border. An accent visible 10+ times per screen is not an accent.

FIX: /design recolor — reserve amber for primary actions only. Let player colors carry more game-screen energy.
P2

Unused Color Token Classes & Missing States

CSS defines utility classes (.c-red, .bg-blue, etc.) that are never used in HTML. Chat has no empty state. Player bar renders nothing when null. The only error feedback is alert().

FIX: /design surface — wire in or remove dead tokens. Add empty/error states for chat, player list, and connection failures.
